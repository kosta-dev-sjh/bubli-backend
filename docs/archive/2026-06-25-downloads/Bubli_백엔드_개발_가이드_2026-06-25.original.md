## 1. 문서 목적

이 문서는 Bubli 백엔드 코드를 개발할 때 팀원이 공통으로 지켜야 하는 규칙을 정리한다.

이 문서의 범위:

- 패키지 구조
- global 공통 구현 기준
- 계층별 책임
- 도메인별 책임
- 도메인 간 연결 규칙
- 명명 규칙
- 권한 검증 규칙
- 데이터 저장 규칙
- 테스트와 API 수동 검증 기준

상세 시나리오, API 명세, DB 상세 설계, 에이전트 세부 설계, 배포 절차는 별도 문서에서 다룬다.

## 2. 개발 기준

Bubli 백엔드는 초기 구현에서 **Spring Boot 기반 모듈형 모놀리식**으로 개발한다.

기본 기준:

```txt
Spring Boot 백엔드 서버 1개
Java
PostgreSQL + pgvector
Spring Data JPA
Spring Security + JWT
Spring AI
WebSocket/STOMP
LocalFileStorage -> S3
```

초기에는 기능별 서버를 분리하지 않는다. 서버 내부 패키지를 도메인 단위로 나누고, 도메인 간 의존성은 공개 Service를 통해 연결한다.

에이전트 기능도 초기에는 백엔드 내부 `agent` 모듈로 구현한다. 단, `agent`는 나중에 별도 실행 단위로 분리할 수 있도록 다른 도메인의 Repository, Entity, 내부 구현 패키지에 직접 의존하지 않는다.

## 3. 협업 규칙

Git 협업 규칙은 `GIT_CONVENTION.md`를 따른다.

핵심 기준:

- `main`, `develop`에 직접 push하지 않는다.
- 작업 브랜치는 `develop`에서 생성한다.
- 기능 개발은 `feature/기능명`을 사용한다.
- 버그 수정은 `fix/버그명`을 사용한다.
- 설정, 문서, 빌드 작업은 `chore/작업명`을 사용한다.
- PR은 팀원 1명 이상 리뷰 후 머지한다.
- 머지 방식은 Squash merge를 사용한다.
- `application-secret.yml`, `.env` 등 민감정보 파일은 커밋하지 않는다.

개발 IDE는 IntelliJ IDEA를 기준으로 한다.

| 패키지         | 기획서 기준    | 역할                                  |
| ----------- | --------- | ----------------------------------- |
| `auth`      | 핵심 업무 모듈  | Google 로그인, 세션, 토큰 재발급, 인증           |

| `memory`    | 기억 모듈     | 프로젝트룸 장기기억, 사용자가 확인한 하루정리 요약        |
| `widget`    | 위젯 모듈     | 버블 표시 데이터, 항목 상태, 사용 집계, 타이머 표시 기준  |

`dashboard`, `notification`, `calendar`, `memo`, `timer`, `suggestion`은 최상위 패키지로 분리하지 않고 `personal` 하위에 둔다.

`task`, `wbs`, `schedule`은 TODO, WBS, 일정 원본에 해당하므로 `work` 하위에 둔다. 개인 TODO와 프로젝트룸 TODO는 모두 `work.task`에서 같은 `tasks` 원본으로 관리하고, `owner_id`, `room_id`, `assignee_id`로 구분한다. 일정은 `schedules` 테이블을 원본으로 하며, 개인 일정과 프로젝트룸 일정은 `room_id` 유무로 구분한다.

`personal`, `work`는 직접 기능을 구현하는 도메인 패키지가 아니라 하위 도메인을 묶는 상위 패키지다. 따라서 `personal/controller`, `personal/service`, `work/controller`, `work/service`, `work/repository`를 기본 구조로 만들지 않는다.

각 도메인 모듈은 아래 구조를 기본으로 한다.
```txt

→ controller / service / dto / type

agent 내부 구현
→ service / job / suggestion / prompt / rag / model 중심
```

```txt
domain
  controller
  service
  repository
  entity
  dto
  type
```

`mapper`는 선택 사항이다. DTO 변환이 단순하면 Service나 정적 팩토리 메서드에서 처리하고, 변환 규칙이 반복되거나 복잡해질 때만 `mapper` 패키지를 둔다.

예시:

```txt
resource
  controller
  service
  repository
  entity
  dto
  type
```

`agent` 모듈은 내부 책임이 많으므로 아래 하위 패키지를 추가로 둘 수 있다.

```txt
agent
  controller
  service
  job
  suggestion
  prompt
  rag
  model
  repository
  entity
  dto
  error
  type
```

`suggestion`, `prompt`, `rag`, `model`, `job`, `error`는 agent 내부 구현 패키지다. 다른 도메인은 이 패키지를 직접 호출하지 않고 `agent.service`의 공개 Service를 통해서만 agent 기능을 요청한다.

`agent.error`에는 에이전트 도메인이 소유하는 `AGENT_*` 에러 코드만 둔다. 실패 응답 형식, 공통 예외 클래스, 전역 예외 처리는 `global.error`의 공통 구조를 사용한다.

`advisor`라는 패키지명은 사용하지 않는다. 화면이나 기획 문구에서 어드바이저, 추천, 제안처럼 표현하더라도 서버 코드에서는 `agent`, `suggestion`, `candidate`처럼 역할이 분명한 이름을 사용한다.

에이전트 분리를 위해 지켜야 할 기준:

- `agent`는 다른 도메인의 Repository를 직접 호출하지 않는다.
- `agent`는 다른 도메인의 Entity를 직접 수정하지 않는다.
- `agent`는 필요한 입력을 공개 Service 요청 DTO로 전달받는다.
- `agent`는 후보 JSON, 분석 결과, 모델 호출 로그까지만 생성하고, 에이전트가 만든 모든 후보는 `agent_suggestions`에 저장한다.
- 확정 데이터 반영은 `resource`, `work`, `personal`, `memory` 등 대상 도메인 Service가 처리한다.

### 4.1 global 패키지 기준

`global`은 모든 도메인에서 공통으로 사용하는 서버 기반 코드만 둔다. 특정 도메인의 비즈니스 규칙은 `global`에 넣지 않는다.

```txt
global
  config
  security
  error
  response
  validation
  trace
  websocket
  type
  util
```

역할:

- `config`: Spring 설정, CORS, WebSocket, OpenAPI, 외부 연동 설정
- `security`: Spring Security, JWT 필터, 인증 사용자 조회
- `error`: 공통 예외, 전역 예외 처리, 에러 코드 매핑
- `response`: 공통 API 응답 형식
- `validation`: 공통 검증 annotation과 validator
- `trace`: 요청 별 traceId 생성과 로그 추적
- `websocket`: WebSocket/STOMP 설정, 인증 인터셉터, topic 구독 권한 검증
- `type`: 여러 도메인에서 같은 의미로 재사용되는 enum
- `util`: 도메인 의미가 없는 순수 유틸리티

`global`에 둘 수 있는 것:

- 인증 사용자 조회 helper
- 공통 응답 wrapper
- 공통 예외와 에러 응답
- traceId 처리
- 공통 날짜, 파일 크기, MIME group 같은 범용 유틸

`global`에 두면 안 되는 것:

- project, room, resource, task 같은 도메인 규칙
- 특정 도메인의 권한 검증 로직
- 특정 도메인의 상태 전이 로직
- 여러 도메인 Repository를 모아서 호출하는 공통 Service
- 모든 상태를 담는 `CommonStatus`, `BaseStatus`, `Status` enum

도메인에서 공통처럼 보이는 코드라도 의미가 특정 도메인에 묶여 있으면 해당 도메인 패키지에 둔다.

### 4.2 global 공통 구현 기준

`global`은 팀 전체 API의 모양과 보안 흐름을 맞추기 위한 공통 기반이다. 도메인 구현을 시작하기 전에 `response`, `error`, `security`, `trace`, `validation` 기준을 먼저 만든다.

#### 4.2.1 response

모든 API 응답은 `ApiResponse<T>`로 감싼다.

```txt
global.response
  ApiResponse<T>
  PageResponse<T>
```

성공 응답:

```json
{
  "success": true,
  "data": {},
  "error": null
}
```

데이터가 없는 성공 응답:

```json
{
  "success": true,
  "data": null,
  "error": null
}
```

목록 응답은 `PageResponse<T>`를 사용한다.

```json
{
  "success": true,
  "data": {
    "items": [],
    "page": 0,
    "size": 20,
    "totalElements": 0,
    "totalPages": 0,
    "hasNext": false
  },
  "error": null
}
```

규칙:

- 성공 응답에는 `message`를 넣지 않는다.
- 생성, 수정, 삭제 성공 후 반환 데이터가 없으면 `data: null`을 허용한다.
- 목록 조회 결과가 비어 있으면 `items: []`를 반환한다.
- Entity를 `data`에 직접 넣지 않는다.
- Controller는 `ApiResponse.success(data)` 또는 `ApiResponse.success(null)` 형태로 반환한다.

#### 4.2.2 error

실패 응답은 `ErrorResponse`로 통일한다.

```txt
global.error
  ErrorCode
  ErrorResponse
  BusinessException
  GlobalExceptionHandler
```

실패 응답:

```json
{
  "success": false,
  "data": null,
  "error": {
    "code": "RESOURCE_404_001",
    "message": "자료를 찾을 수 없습니다.",
    "traceId": "..."
  }
}
```

Validation 실패 응답:

```json
{
  "success": false,
  "data": null,
  "error": {
    "code": "COMMON_400_002",
    "message": "요청 값이 올바르지 않습니다.",
    "traceId": "...",
    "fields": [
      {
        "field": "title",
        "reason": "제목은 필수입니다."
      }
    ]
  }
}
```

`ErrorCode`는 HTTP status, 내부 code, 기본 message를 가지는 공통 계약이다. 공통 에러 코드는 `global.error`에서 관리하고, agent 전용 에러 코드는 `agent.error.AgentErrorCode`에서 관리한다.

```txt
ErrorCode
  httpStatus
  code
  message
```

```txt
agent.error
  AgentErrorCode
```

에러 코드 prefix:

|prefix|대상|
|---|---|
|`COMMON`|공통 요청, validation, 서버 오류|
|`AUTH`|로그인, 토큰, 인증|
|`USER`|사용자, 설정, 동의|
|`PROJECT`|프로젝트룸, 초대, 멤버|
|`RESOURCE`|자료, 파일, 다운로드, 분석 상태|
|`WORK`|TODO, WBS, 내부 일정|
|`CHAT`|1:1 채팅, 프로젝트룸 채팅|
|`AGENT`|에이전트 작업, 후보, 모델 호출|
|`MEMORY`|프로젝트룸 장기기억, 하루정리 요약|
|`WIDGET`|버블 항목 상태, 위젯 사용 집계|
|`VOICE`|LiveKit 방, 토큰, 참가자|
|`LOCALSYNC`|로컬 폴더 동기화, outbox|
|`STORAGE`|LocalFileStorage, S3, 파일 검증|

`AGENT_*` 에러 코드는 에이전트 후보 생성, agent job, 모델 호출, Structured Output 검증처럼 agent 모듈이 책임지는 실패에 사용한다. `AGENT_*` 에러 코드 정의는 `agent.error.AgentErrorCode`에 둔다. 이후 agent 서버를 분리하면 이 정의를 agent 서버 쪽으로 이동하고, API 서버는 agent 서버가 반환한 에러 code를 공통 응답 형식으로 전달하거나 변환한다.

코드 형식:

```txt
{PREFIX}_{HTTP_STATUS}_{SEQUENCE}

예:
COMMON_400_001
AUTH_401_001
PROJECT_403_001
RESOURCE_404_001
CHAT_409_001
AGENT_422_001
```

기본 HTTP status 기준:

|HTTP status|사용 기준|
|---|---|
|400|요청 형식이나 파라미터가 잘못됨|
|401|로그인이 필요하거나 토큰이 유효하지 않음|
|403|로그인했지만 접근 권한이 없음|
|404|대상이 존재하지 않음|
|409|현재 상태와 충돌하거나 중복 요청임|
|422|요청은 이해했지만 도메인 규칙이나 구조화 결과 검증에 실패함|
|500|예상하지 못한 서버 오류|

예외 규칙:

- Service는 예상 가능한 비즈니스 실패를 `BusinessException(ErrorCode)`로 던진다.
- Controller에서 try-catch로 비즈니스 예외를 처리하지 않는다.
- `GlobalExceptionHandler`가 모든 예외를 공통 실패 응답으로 변환한다.
- 예상하지 못한 예외는 `COMMON_500_001`로 응답한다.
- 사용자에게 보여줄 수 없는 내부 오류 내용은 `message`에 넣지 않는다.

#### 4.2.3 security

인증은 `global.security`에서 처리하고, 도메인 권한 검증은 각 도메인 Service에서 처리한다.

```txt
global.security
  SecurityConfig
  JwtAuthenticationFilter
  JwtTokenProvider
  AuthUser
  CurrentUser
  CurrentUserArgumentResolver
```

Controller는 로그인 사용자를 `@CurrentUser AuthUser`로 받는다.

```java
@GetMapping("/api/project-rooms/{roomId}")
public ApiResponse<ProjectRoomResponse> getRoom(
    @CurrentUser AuthUser authUser,
    @PathVariable UUID roomId
) {
    return ApiResponse.success(projectRoomService.getRoom(authUser.userId(), roomId));
}
```

`AuthUser`는 최소 인증 정보만 가진다.

```java
public record AuthUser(
    UUID userId,
    String bubliId
) {}
```

규칙:

- request body로 `userId`를 받지 않는다.
- 로그인 사용자 식별은 JWT에서 가져온다.
- JWT secret은 서버 설정으로만 관리하고 프론트, Tauri 앱, 공개 사이트에 노출하지 않는다.
- 인증 실패는 `AUTH_401_001` 계열로 응답한다.
- 권한 실패는 각 도메인 Service에서 검증하고 `403` 계열 에러로 응답한다.
- 프로젝트룸 초대는 기존 회원 대상 초대만 처리한다. 링크 초대는 초기 서버 범위에서 제외한다.

도메인 권한 검증 위치:

|대상|검증 위치|기준|
|---|---|---|
|프로젝트룸|`project.service`|`room_members`|
|자료|`resource.service`|`owner_id`, `visibility`, `room_members`|
|TODO/WBS/일정|`work.task.service`, `work.wbs.service`, `work.schedule.service`|`owner_id`, `assignee_id`, `room_members`|
|채팅|`chat.service`|`chat_room_members`|
|보이스|`voice.service`|`room_members`|
|위젯|`widget.service`|본인 `userId`, `widget_context_settings.selected_room_id`가 있으면 접근 가능한 프로젝트룸|
|에이전트|`agent` 호출 전 대상 도메인 Service|자료, 프로젝트룸, 채팅 권한|

#### 4.2.4 trace

모든 요청은 `traceId`를 가진다.

```txt
global.trace
  TraceIdFilter
  TraceIdHolder
```

규칙:

- 요청 시작 시 `traceId`를 생성한다.
- 요청 header에 `X-Trace-Id`가 있으면 검증 후 사용한다.
- `traceId`는 MDC에 저장해 로그에 함께 남긴다.
- 응답 header에 `X-Trace-Id`를 포함한다.
- 실패 응답의 `error.traceId`에 같은 값을 넣는다.
- 비동기 작업이나 에이전트 job을 만들 때 요청 `traceId`를 job/event 로그에 연결할 수 있게 전달한다.

로그 금지 값:

- 비밀번호
- JWT secret
- 모델 API key
- LiveKit secret
- S3 secret
- 문서 원문 전체
- 개인 에이전트 원문 대화
- 화면 전체 내용과 키보드 입력

#### 4.2.5 validation

요청 값 검증은 DTO와 공통 validation으로 처리한다.

```txt
global.validation
  ValidationError
  custom annotations
  validators
```

규칙:

- Controller Request DTO에는 필요한 Bean Validation annotation을 붙인다.
- validation 실패는 `COMMON_400_002`와 `fields` 배열로 응답한다.
- 도메인 상태 전이 검증은 validation annotation이 아니라 Service에서 처리한다.
- 파일 크기, MIME group 같은 범용 검증은 `global.validation` 또는 `global.util`에 둘 수 있다.
- roomId 접근 가능 여부, resource visibility 같은 도메인 권한 검증은 `global.validation`에 두지 않는다.

## 5. 계층별 규칙

### 5.1 Controller

Controller는 HTTP 요청과 응답만 담당한다.

- 요청 DTO를 받는다.
- 로그인 사용자 정보를 가져온다.
- Service를 호출한다.
- 공통 응답 형식으로 반환한다.
- Entity를 직접 반환하지 않는다.
- 권한, 상태 변경, DB 접근 로직을 작성하지 않는다.

### 5.2 Service

Service는 비즈니스 흐름을 담당한다.

- 권한을 검증한다.
- 상태 전이를 검증한다.
- 트랜잭션 경계를 관리한다.
- Repository를 호출한다.
- 다른 도메인 데이터가 필요하면 해당 도메인의 공개 Service를 호출한다.
- Controller로 Entity를 직접 노출하지 않는다.

### 5.3 Repository

Repository는 DB 접근만 담당한다.

- Spring Data JPA Repository를 기본으로 사용한다.
- 단순 조건 조회는 Query Method로 작성한다.
- 고정된 조건의 명확한 조회는 JPQL `@Query`로 작성한다.
- 동적 조건, 여러 테이블 조인, 검색 필터가 많은 조회는 QueryDSL 또는 Custom Repository로 분리한다.
- 다른 도메인의 Repository를 직접 호출하지 않는다.

### 5.4 Entity

Entity는 DB 테이블과 상태 변경 규칙을 표현한다.

- 상태값은 enum으로 관리한다.
- enum은 각 도메인의 `type` 패키지에 둔다.
- 여러 도메인에서 같은 의미로 재사용되는 enum만 `global/type`에 둔다.
- Entity에는 최소한의 상태 변경 메서드만 둔다.
- API 응답 전용 필드는 Entity에 넣지 않는다.
- 연관관계는 필요한 범위에서만 설정한다.

공통 enum 후보:

```txt
global/type
  LanguageCode.java
  ThemeMode.java
  SortDirection.java
  FileMimeGroup.java
```

금지:

- 모든 상태를 담는 `CommonStatus`, `BaseStatus`, `Status` enum을 만들지 않는다.
- 의미와 상태 전이가 다르면 도메인별 enum으로 분리한다.
- DB에는 정의된 enum 상수만 저장하고 임의 문자열을 저장하지 않는다.

### 5.5 DTO

API DTO는 Controller의 요청과 응답에만 사용한다.

- Request와 Response를 분리한다.
- Entity를 그대로 반환하지 않는다.
- 프론트에 필요한 값만 포함한다.
- 내부 서비스 간 전달 값은 Command, Query, Context, Result 객체로 분리한다.

이렇게 분리하는 이유는 Service나 agent 모듈에 넘길 값을 호출하는 쪽에서 미리 정리해서 전달하기 위해서다. 예를 들어 `resource`가 agent에게 자료 분석을 요청할 때는 `Resource` Entity 전체를 넘기지 않고, 권한 확인이 끝난 뒤 agent가 판단에 필요한 `resourceId`, `roomId`, `visibility`, `fileName`, `extractedText` 같은 값만 Context나 Command로 묶어 전달한다.

즉 Command, Query, Context, Result는 다른 계층이나 다른 도메인에 일을 부탁할 때 사용하는 전달용 값 객체다. agent 연결 값은 나중에 내부 API 요청/응답으로 전환될 수 있으므로, 처음부터 Entity나 MultipartFile 같은 서버 내부 구현 세부사항을 넣지 않고 agent가 실제로 판단에 필요한 값만 담는다.

DTO와 내부 전달 객체 이름:

|종류|용도|예시|
|---|---|---|
|`Request`|Controller가 받는 API 요청|`CreateProjectRoomRequest`|
|`Response`|Controller가 반환하는 API 응답 데이터|`ProjectRoomResponse`|
|`Command`|쓰기 Service에 전달하는 입력|`CreateProjectRoomCommand`|
|`Query`|조회 Service에 전달하는 조건|`ProjectRoomListQuery`|
|`Context`|다른 도메인이나 agent에 넘기는 제한된 맥락|`ResourceAnalysisContext`|
|`Result`|Service가 Controller 또는 다른 Service에 반환하는 결과|`ProjectRoomResult`|

규칙:

- Request DTO를 Service까지 그대로 넘기지 않는다.
- Service는 Controller나 HTTP를 몰라야 한다.
- Command, Query, Context, Result는 DTO의 한 종류로 본다.
- Command, Query, Context, Result는 각 도메인의 `dto` 또는 `service` 하위에 둘 수 있으며, Controller Request/Response와 재사용하지 않는다.
- 현재 패키지 구조를 유지하기 위해 `command`, `query`, `context`, `result` 하위 패키지를 필수로 만들지 않는다.
- 객체 수가 많아져 한 패키지 안에서 구분이 어려워질 때만 하위 패키지 분리를 검토한다.
- 외부 API, Tauri IPC, agent 내부 입력값은 화면 API DTO와 분리한다.
- agent와 연결되는 값 객체는 나중에 내부 API 요청/응답으로 전환될 수 있도록 Entity, Repository, HttpServletRequest, MultipartFile 같은 구현 세부사항을 포함하지 않는다.
- Boolean 필드는 Java에서 `enabled`, `visible`, `archived`처럼 의미 중심으로 이름 짓고, JSON 필드도 같은 의미를 유지한다.

### 5.6 Transaction

트랜잭션은 Service 계층에서 관리한다.

규칙:

- 쓰기 Service 메서드는 `@Transactional`을 사용한다.
- 조회 Service 메서드는 `@Transactional(readOnly = true)`를 사용한다.
- Controller와 Repository에 트랜잭션 경계를 두지 않는다.
- 외부 API 호출, S3 업로드, LiveKit 토큰 발급, 모델 호출을 DB 트랜잭션 안에 오래 묶지 않는다.
- DB 저장 후 외부 작업이 필요한 경우 상태값을 먼저 저장하고 재시도 또는 보상 처리 기준을 둔다.
- agent 실행처럼 오래 걸리는 작업은 `agent_jobs` 상태를 저장한 뒤 비동기로 처리한다.
- 채팅 메시지는 DB 저장 성공 후 WebSocket으로 전송한다.
- `local_sync_outbox`, `client_message_id`, `rollup_key`처럼 중복 방지 키가 있는 요청은 같은 키로 여러 번 들어와도 같은 결과가 되도록 처리한다.

## 6. 명명 및 응답 규칙

API JSON 필드와 Java 필드는 `camelCase`를 사용한다.
DB 테이블명과 컬럼명은 `snake_case`를 사용한다.
enum 상수는 `UPPER_SNAKE_CASE`를 사용한다.

예시:

|구분|예시|
|---|---|
|API JSON / Java 필드|`clientMessageId`, `roomSequence`, `lastReadSequence`|
|DB 컬럼|`client_message_id`, `room_sequence`, `last_read_sequence`|
|DB 테이블|`chat_messages`, `room_members`|
|enum 상수|`ROOM_SHARED`, `AGENT_RESPONSE`|

Java 코드는 아래 기준을 따른다.

```txt
Class, DTO, Entity, Enum: PascalCase
field, method, variable: camelCase
enum constant: UPPER_SNAKE_CASE
package: lower_case
```

메서드 이름 기준:

|계층|명명 기준|예시|
|---|---|---|
|Controller|HTTP 동작보다 업무 의미를 드러낸다|`createProjectRoom`, `getResourceDetail`|
|Service 쓰기|명령형 동사를 사용한다|`create`, `update`, `approve`, `revoke`, `sync`|
|Service 조회|`get`으로 통일한다|`getResourceDetail`, `getProjectRooms`, `getDashboardTasks`|
|Repository|Spring Data JPA 규칙을 따른다|`findByRoomIdAndStatus`, `existsByEmail`|

Service 조회 메서드는 단건, 목록, 조건 검색 모두 `get`을 사용한다.
`find`, `search`, `list`는 Service 메서드명으로 쓰지 않는다.
Repository는 Spring Data JPA 관례를 따르므로 `findBy...`, `existsBy...`, `countBy...`를 사용할 수 있다.

필드 이름 기준:

- DB 컬럼은 데이터 딕셔너리를 기준으로 만든다.
- Entity와 DTO 필드는 DB 컬럼의 `snake_case`를 Java `camelCase`로 변환해 사용한다.
- API request/response JSON 필드는 Java DTO와 같은 `camelCase`를 사용한다.
- 예: `created_at` -> `createdAt`, `room_id` -> `roomId`
- ID 필드는 Java에서 `userId`, `roomId`, `resourceId`를 사용하고 DB에서는 `user_id`, `room_id`, `resource_id`를 사용한다.

클래스 이름 기준:

```txt
CreateResourceRequest
CreateResourceCommand
ResourceDetailResponse
ResourceResult
ResourceStatus
ResourceVisibility
```

API 성공 응답은 `ApiResponse<T>`를 사용한다.

```json
{
  "success": true,
  "data": {},
  "error": null
}
```

실패 응답:

```json
{
  "success": false,
  "data": null,
  "error": {
    "code": "RESOURCE_404_001",
    "message": "자료를 찾을 수 없습니다.",
    "traceId": "..."
  }
}
```

규칙:

- 목록 응답은 `PageResponse<T>`를 `ApiResponse`의 `data`에 담아 반환한다.
- 성공 응답에는 `message`를 넣지 않는다.
- 반환 데이터가 없는 성공 응답은 `data: null`을 허용한다.
- 모든 실패 응답에는 `error.code`, `error.message`, `error.traceId`를 포함한다.
- 실패 응답은 `ErrorResponse`를 사용한다.
- Entity를 API 응답으로 직접 반환하지 않는다.
- API 경로는 기획서의 `/api/...` 구조를 따른다.

모든 주요 API는 로그인 사용자 기준으로 권한을 확인한다.

기본 권한 기준:

|대상|권한 기준|

## 8. 도메인별 책임

### 8.1 auth

- Google OAuth/OIDC 로그인
- `users.google_sub` 기준 사용자 식별
- JWT 발급
- refresh token과 서버 세션 관리
- 토큰 만료 처리
- 로그아웃과 세션 만료 처리
- `user_sessions` 관리

email/password 회원가입과 비밀번호 암호화는 초기 인증 기준에 포함하지 않는다. 인증은 Google 로그인으로 사용자를 확인하고, 서버는 사용자 세션과 refresh token 수명 주기를 관리한다.

auth는 프로젝트룸, 자료, 작업 도메인 로직을 직접 처리하지 않는다.

### 8.2 user

- 내 프로필
- 사용자별 설정
- 알림 설정
- 개인정보 동의
- 기본 프로젝트룸 설정
- Bubli ID 기준 사용자 검색
- 친구 요청 보내기
- 친구 요청 수락
- 친구 제거
- `friend_requests`, `friendships` 관리

사용자별 설정은 프로젝트룸 설정과 분리한다.

친구 기능은 별도 최상위 도메인으로 분리하지 않고 `user` 도메인에서 관리한다. 친구 차단 기능은 초기 구현 범위에 포함하지 않는다.

친구 관계 사용 기준:

- `user`는 Bubli ID 검색, 친구 요청, 친구 수락, 친구 제거를 처리한다.
- `project`는 기존 회원을 프로젝트룸에 초대할 때 사용자 식별 결과를 사용한다. 친구 관계가 필요한 정책은 별도로 확정된 경우에만 검증한다.
- `chat`은 ACCEPTED 친구 관계를 기준으로 1:1 채팅방 생성과 조회를 처리한다.
- `project`와 `chat`은 친구 요청 상태를 직접 변경하지 않는다.

### 8.3 project

- 프로젝트룸 CRUD
- 서비스상 프로젝트 생성, 수정, 종료 처리는 `project_rooms` 기준으로 처리
- 프로젝트룸 기본 정보 관리
- 계약 금액, 입금 상태, 입금 예정일, 실제 입금일 같은 계약/입금 참고값 관리
- room_members 관리
- 기존 회원 프로젝트룸 초대
- 멤버 역할 변경
- 멤버 나가기, 내보내기
- 프로젝트룸 이벤트 저장

별도 `projects` 테이블은 두지 않는다. 서비스에서 말하는 프로젝트 단위는 서버 DB의 `project_rooms` 한 row와 같다.
백엔드 API와 Service에서는 프로젝트와 프로젝트룸 CRUD를 분리하지 않고, `project_rooms` 기준으로 생성, 수정, 종료 처리를 한다.
프로젝트룸 접근 권한의 기준은 `room_members`다.
`project_rooms`는 단순한 채팅방이나 협업방이 아니라 서비스에서 말하는 프로젝트 자체다. 따라서 프로젝트 이름, 고객명, 설명, 시작/종료 시각뿐 아니라 계약 금액, 입금 상태, 입금 예정일, 실제 입금일처럼 프로젝트 운영에 필요한 계약/입금 참고값도 함께 관리한다.
`invitations`는 기존 회원을 프로젝트룸에 초대하는 용도로 사용한다. 링크 초대는 초기 서버 범위에서 제외한다.
프로젝트룸 단위 변경 이벤트는 `project_room_events`에 저장한다. WebSocket 전송, 누락 이벤트 보충, 알림, 활동 로그가 필요할 때 이 이벤트 기록을 기준으로 연결한다.

### 8.4 resource

- 개인 자료 업로드
- 프로젝트룸 자료 업로드
- 자료 목록과 상세 조회
- 파일 메타데이터 관리
- 자료 버전 관리
- 자료 댓글
- 다운로드 권한 검증
- 텍스트 추출 요청
- 자료 분석 상태 관리
- 자료 단위 요약 관리

resource는 파일 원본과 자료 접근 권한을 책임진다.
자료 단위 요약은 `resource_summaries`에 저장한다. AI가 읽을 만한 문서 분류와 분석 상태는 `ai_documents`로 관리하고, 승인/보류/거절이 필요한 후보는 `agent_suggestions`에 저장한다.

에이전트가 자료 분석을 수행하더라도 자료 접근 권한, 파일 원본 위치, 텍스트 추출 결과 제공 범위는 `resource`가 결정한다. `agent`는 `resources`, 파일 저장소, 다운로드 URL을 직접 조회하거나 생성하지 않고, `resource`가 권한 확인 후 전달한 제한된 context만 사용한다.

### 8.5 agent

- 에이전트 작업 요청 접수
- `resources` 자료 중 AI 분석 대상 문서 분류
- AI 분석 대상 문서를 `ai_documents`에 저장
- `ai_documents.resource_id`에 포함된 자료만 임베딩 대상으로 사용
- 문서 분석, 계약 검토, 요구사항, WBS, TODO, 확인 질문, 문서 초안 후보 생성
- 추천, 제안 성격의 후보 생성과 관리
- 프로젝트룸 채팅 명령어 응답 생성
- 자료 요약과 프로젝트룸 맥락 유지에 필요한 요약 후보 생성
- Structured Output 검증

agent는 확정 데이터를 직접 만들지 않는다. 화면에서 추천, 제안, 후보로 보이는 값은 서버에서 agent가 만든 후보 데이터로 관리한다.
에이전트가 생성한 후보는 종류와 출처가 달라도 모두 `agent_suggestions`에 저장한다. 자료 기반 제안은 `agent_suggestions.resource_id`로 원본 자료를 연결하고, `suggestion_type`으로 REQUIREMENT, WBS, TODO, SCHEDULE, QUESTION, CONTRACT_FIELD, CONTRACT_REVIEW, DOCUMENT_DRAFT, DAILY_SUMMARY, MEMO 같은 후보 종류를 구분한다. 후보 본문, 원문 근거, 대상 도메인, field_key 같은 세부 정보는 `payload_json`에 담는다.
사용자가 승인하기 전에는 실제 TODO, WBS, 일정, 메모, 계약 참고값, 장기기억으로 저장하지 않는다. 사용자가 승인하면 `work`, `personal`, `resource`, `memory`, `project` 같은 대상 도메인 모듈이 자신의 확정 데이터로 저장한다.

agent는 판단과 생성에 집중한다. 원본 데이터 조회, 권한 검증, 확정 데이터 저장은 대상 도메인의 책임이다.
자료 분석 흐름은 `resources` 원본을 직접 분석 대상으로 고정하지 않고, 먼저 AI가 읽을 만한 문서인지 분류한 뒤 필요한 자료만 `ai_documents`에 저장한다. 이후 임베딩은 `ai_documents.resource_id`에 포함된 `resources`만 대상으로 하며, 실제 chunk와 embedding은 원본 `resource_id` 기준으로 `resource_embeddings`에 저장한다.

`agent.suggestion`은 문서, 채팅, 댓글, 메모, 활동 맥락처럼 여러 출처에서 만들어진 모든 에이전트 후보를 저장하고 상태를 관리하는 영역이다. 요구사항, TODO, WBS, 확인 질문, 계약 추출값, 문서 검토 항목, 문서 초안, 하루정리, 메모 제안은 별도 후보 테이블로 나누지 않고 `agent_suggestions` 한 곳에 저장한다. 단, 승인된 후보를 실제 TODO, WBS, 일정, 메모, 계약 참고값, 요약 데이터로 바꾸는 책임은 대상 도메인이 가진다.

agent가 받을 수 있는 입력:

- 대상 도메인 Service가 권한 확인 후 만든 요청 DTO
- 자료 텍스트 추출 결과 중 분석에 필요한 범위
- 프로젝트룸 역할, 자료 공개 범위, 채팅 맥락처럼 이미 검증된 context
- Tauri 로컬에서 사용자가 승인해 전달한 개인 에이전트 context

agent가 직접 가져오면 안 되는 입력:

- 다른 도메인의 Entity
- 다른 도메인의 Repository 조회 결과
- 권한 확인 전의 resourceId, roomId, filePath만 받은 뒤 agent 내부에서 직접 조회하는 흐름
- 프로젝트룸 에이전트에서 PERSONAL 자료나 개인 에이전트 원문 대화

agent가 반환할 수 있는 결과:

- AI 문서 분류 결과
- 임베딩 생성 결과
- 후보 JSON
- `agent_suggestions` 저장 결과
- 분석 결과
- 확인 질문
- 채팅 응답 후보
- 모델 호출 상태와 실패 사유

agent가 직접 확정하면 안 되는 결과:

- TODO, WBS, 일정, 메모, 장기기억, 자료 분석 확정값
- 파일 삭제, 파일 공유, 외부 전송 같은 사용자 데이터 변경
- 계약 문제 여부나 요구사항 확정 판단

agent 에러 처리 기준:

- agent 후보 생성, agent job, 모델 호출, Structured Output 검증 실패는 `AGENT_*` 에러로 처리한다.
- 자료 없음, 자료 권한 없음, 프로젝트룸 접근 불가처럼 agent 호출 전에 확인해야 하는 문제는 `RESOURCE_*`, `PROJECT_*`, `CHAT_*` 같은 대상 도메인 에러로 처리한다.
- agent 후보 승인 후 TODO, WBS, 일정, 메모, memory로 확정 저장하는 과정의 실패는 `WORK_*`, `PERSONAL_*`, `MEMORY_*` 같은 target 도메인 에러로 처리한다.
- `AGENT_*` 에러 코드 정의는 `agent.error.AgentErrorCode`에 둔다.
- 실패 응답 형식, 공통 예외 클래스, 전역 예외 처리는 `global.error`를 사용한다.
- agent 서버 분리 시점에는 `AGENT_*` 에러 코드 정의를 agent 서버 쪽으로 옮기고, API 서버는 agent 서버의 에러 code를 전달하거나 공통 응답 형식으로 변환한다.

금지:

- `tasks` 직접 생성
- `wbs_items` 직접 생성
- `schedules` 직접 생성
- 파일 삭제
- 파일 공유
- 계약 문제 여부 단정

### 8.6 personal

- 사용자 기준 대시보드 조합 조회
- 개인 메모와 프로젝트룸 메모 화면 처리
- 타이머 시작, 일시정지, 종료
- 작업 시간 기록
- 사용자 알림 저장과 조회
- 개인 캘린더 화면 구성과 일정 조회 결과 표시
- 개인 제안 표시와 확인 상태 관리

personal은 사용자 개인 기준 화면과 기능을 책임진다. 개인 TODO, 프로젝트룸 TODO, WBS, 일정이 대시보드에 표시되더라도 TODO/WBS 원본은 `work`, 일정 원본은 `schedules`가 소유한다. personal.dashboard는 `work`의 조회 결과, 메모/타이머/알림, 일정 조회 결과를 사용자 기준으로 조합한다.

personal.calendar는 개인 캘린더 화면을 구성하고 일정을 표시하는 소비자 역할이다. Google Calendar에서 가져온 일정은 외부 일정 레이어로 화면에 표시하고, Bubli에서 생성한 일정은 `schedules`에 저장한 뒤 같은 캘린더 화면 위에 추가로 표시한다. Bubli 일정 중 개인 일정은 `room_id`가 없고 프로젝트룸 일정은 `room_id`가 있다. `google_event_id`는 Google에서 가져온 외부 일정과 중복 표시를 막거나 출처를 식별해야 할 때 사용한다. 패키지 구조는 MVP 단순화를 위해 `work.schedule`을 유지하지만, `personal.calendar`는 일정 원본을 직접 소유하거나 임의로 생성/수정하지 않는다.

`memos`는 개인 메모와 프로젝트룸 메모를 함께 담을 수 있다. 개인 메모는 `user_id` 기준으로, 프로젝트룸 메모는 `room_id`와 `room_members` 권한 기준으로 처리한다. 자료에 대한 댓글은 메모가 아니라 `resource_comments`로 분리한다.

`time_logs`는 `timer_type`으로 일반 타이머와 작업 타이머를 구분한다. 작업 타이머는 프로젝트룸과 연결될 수 있고, 위젯 로컬 타이머 상태는 `widget_context_settings.selected_room_id`와 함께 해석한다.

개인 에이전트가 하루정리, 개인 제안, 메모 후보를 만들더라도 후보 원본은 `agent_suggestions`에 저장하고, 개인 화면의 확정 저장 책임은 `personal` 또는 대상 원본 도메인이 가진다. `personal`은 agent 후보를 사용자에게 보여주고 확인 상태를 관리할 수 있지만, agent 결과를 사용자 승인 없이 메모나 일정으로 확정 저장하지 않는다.

하위 패키지:

```txt
personal
  dashboard
  memo
  timer
  notification
  calendar
  suggestion
```

### 8.7 work

- 개인 TODO CRUD
- 프로젝트룸 TODO CRUD
- WBS CRUD
- WBS 순서 변경
- 작업 상태 변경
- 일정 CRUD
- task, wbs_item, room과 연결된 일정 관리
- 승인된 후보를 실제 TODO/WBS/일정으로 반영
- 대시보드와 위젯 조합에 필요한 TODO/WBS/일정 조회 결과 제공

work는 TODO, WBS, 일정 저장 흐름을 책임진다. 개인 TODO와 프로젝트룸 TODO는 같은 `tasks` 원본을 사용하고, `owner_id`, `room_id`, `assignee_id`로 구분한다.
work는 대시보드나 위젯 화면을 직접 소유하지 않고, 해당 화면에 필요한 업무 원본 조회 결과를 제공한다.

일정은 `schedules` 테이블을 기준으로 통합 관리한다. 개인 일정은 `room_id`가 없는 일정이고, 프로젝트룸 일정은 `room_id`가 있는 일정이다. Google Calendar 일정은 캘린더 화면에 함께 보여줄 외부 일정으로 가져오고, Bubli에서 생성한 일정은 `schedules`에 저장해 Google Calendar 일정 위에 추가로 표시한다. 패키지 구조는 유지하되, `work.schedule`은 `schedules`를 기준으로 Bubli 일정 생성, 수정, 삭제를 처리하는 모듈로 본다.

에이전트가 TODO, WBS, 일정 후보를 만들 수 있지만 후보 원본은 `agent_suggestions`에 저장하고, `tasks`, `wbs_items`, `schedules` 확정 생성은 `work`가 처리한다. 일정 후보는 `agent_suggestions.suggestion_type = SCHEDULE`로 저장하고, 사용자가 승인한 뒤 `work.schedule`의 Service가 `room_id` 유무에 따라 개인 일정 또는 프로젝트룸 일정으로 `schedules`에 반영한다. 승인된 Bubli 일정은 Google Calendar에 바로 쓰는 것이 아니라, 캘린더 화면에서 Google Calendar에서 가져온 일정과 함께 표시한다.

하위 패키지:

```txt
work
  task
  wbs
  schedule
```

### 8.8 chat

- 친구와의 1:1 채팅방
- 사용자 간 프로젝트룸 채팅방
- 메시지 저장
- `client_message_id` 중복 방지
- `room_sequence` 발급
- WebSocket 전달
- 프로젝트룸 에이전트 명령어 수신 시 agent 공개 Service 호출
- 프로젝트룸 agent 응답 결과를 `AGENT_RESPONSE` 메시지로 저장

채팅 원본은 서버 DB의 `chat_messages`다.

친구와의 1:1 채팅은 사용자 간 메시지이므로 chat 모듈이 처리하며, 에이전트를 호출하지 않는다.

개인 에이전트 대화는 agent 모듈의 개인 작업 흐름으로 처리하며, 서버 DB에 원문 대화를 저장하지 않는다.

프로젝트룸 채팅에서는 사용자가 에이전트를 호출할 수 있다. 이 경우 chat 모듈이 명령어 메시지를 받은 뒤 agent 공개 Service를 호출하고, agent 응답 결과를 `AGENT_RESPONSE` 메시지로 저장해 같은 프로젝트룸 채팅방에 표시한다.

프로젝트룸 채팅에서 agent 호출 전 권한 확인과 조회 범위 제한은 `chat`과 `project`가 담당한다. agent는 전체 채팅방 원문을 직접 조회하지 않고, `chat`이 제한한 최근 메시지, room context, memory summary만 입력으로 받는다.

### 8.9 memory

- 프로젝트룸 장기기억 저장
- 사용자가 확인한 하루정리 요약 저장
- `room_memory_summaries` 관리
- `daily_summaries` 관리

memory는 에이전트가 만든 요약 후보를 그대로 확정하지 않는다. 사용자가 확인한 결과만 서버의 요약 데이터로 저장한다.

개인 에이전트 원문 대화는 memory에 저장하지 않는다.

agent가 요약 후보를 생성하더라도 `room_memory_summaries`, `daily_summaries` 저장 책임은 `memory`가 가진다. memory는 후보의 출처, 승인 여부, 사용자 수정 여부를 구분할 수 있게 저장 흐름을 설계한다.

### 8.10 widget

- 버블 표시 데이터 조합
- 위젯 현재 프로젝트룸 context 관리
- 버블별 활성 상태, 위치, 크기, 표시 옵션 관리
- 버블 항목 상태 관리
- 위젯 사용 집계 관리
- 타이머 표시 기준 관리
- 각 버블에 필요한 summary 조합
- 접근 가능한 프로젝트룸 정보 요약

widget이 직접 소유하는 서버 데이터는 `widget_context_settings`, `widget_bubble_settings`, `widget_item_states`, `widget_daily_summaries`다.

`widget_context_settings.selected_room_id`는 위젯이 현재 바라보는 프로젝트룸 context다. 값이 없으면 개인 모드이고, 값이 있으면 해당 프로젝트룸 기준으로 TODO, 일정, 메모, 타이머 요약을 조회한다. 선택된 room은 반드시 `room_members` 접근 권한을 확인해야 한다.

`widget_bubble_settings`는 기존 `widget_bubbles`, `widget_layouts`, `widget_preferences` 역할을 합친 버블 설정 원본으로 본다. 버블별 활성 상태, 위치, 크기, 표시 옵션은 이 테이블 기준으로 관리한다.

widget이 직접 소유하지 않는 데이터:

- TODO 원본: work.task
- WBS 원본: work
- 일정 원본: schedules / work.schedule
- 캘린더 화면 조합: personal.calendar
- 메모 원본: memos
- 타이머와 시간 기록 원본: time_logs
- 에이전트 제안 원본: agent
- 자료 원본: resource
- 채팅 원본: chat
- 알림 원본: personal.notification

widget은 원본 데이터를 복사 저장하지 않고 각 도메인의 조회 결과를 사용자 기준으로 조합한다.
위젯의 프로젝트룸 context는 요청마다 임의로 받은 `roomId`가 아니라 `widget_context_settings.selected_room_id`를 기준으로 해석한다. `selected_room_id`가 있으면 widget은 `room_members` 기준으로 접근 가능 여부를 검증한 뒤 해당 프로젝트룸 데이터만 조합한다.

### 8.11 voice

- LiveKit room 생성
- LiveKit 참가 토큰 발급
- 참가자 상태 저장
- 통화 종료 처리

LiveKit API key와 secret은 서버에만 둔다.
voice는 LiveKit 연동과 참가자 상태를 담당한다. 참가 권한은 `room_members`와 채팅방 접근 권한을 기준으로 확인한다.

보이스챗은 별도 채팅 도메인이 아니라 1:1 채팅 또는 프로젝트룸 채팅 안에서 사용할 수 있는 실시간 음성 기술이다. 초기 구현은 프로젝트룸 보이스를 우선하고, 녹음과 음성 요약은 향후 확장으로 둔다.

### 8.12 localsync

- Tauri 개인 관리 폴더 이벤트 수신
- 로컬 파일 색인 동기화
- 서버 개인 자료함 반영
- 동기화 대기열 처리
- 중복 요청 방지용 `idempotency_key` 검증

localsync는 사용자가 개인 자료함 동기화를 켠 관리 폴더 이벤트를 서버 개인 자료함에 자동 반영한다. 개인 자료를 프로젝트룸에 자동 공유하지 않는다.

### 8.13 storage

- 파일 저장 인터페이스
- LocalFileStorage 구현
- S3Storage 구현
- 다운로드 URL 발급
- 파일 크기 제한
- 저장 용량 계산에 필요한 파일 메타데이터 제공

개발 초반은 LocalFileStorage로 검증할 수 있지만, 운영 기준은 S3다.

### 8.14 activity

- 활동 감지 동의 상태 확인
- 현재 사용 앱 이름 기록
- 창 제목 기반 작업 맥락 기록
- 머문 시간 기록
- 오늘 작업 맥락 조회
- 활동 기록 삭제
- 개인 하루정리와 에이전트 제안에 필요한 작업 맥락 제공

activity는 사용자의 동의가 켜진 경우에만 기록한다.

저장 가능한 값:

- 앱 이름
- 창 제목
- 머문 시간

저장하지 않는 값:

- 화면 전체 내용
- 키보드 입력
- 파일 원문
- 개인 에이전트 원문 대화

## 9. 도메인 연결 규칙

다른 도메인의 Entity와 Repository를 직접 사용하지 않는다.

허용:

```txt
controller -> same_domain.service
same_domain.service -> same_domain.repository
domain.service -> other_domain.public_service
```

금지:

```txt
controller -> repository
domain.service -> other_domain.repository
domain.service -> other_domain.entity 직접 변경
domain.repository -> other_domain.repository
```

공개 Service 기준:

- 다른 도메인에서 호출할 수 있는 메서드는 Service에 명시적으로 둔다.
- 공개 Service는 Entity를 반환하지 않고 Result 또는 응답 전용 객체를 반환한다.
- 공개 Service 메서드는 호출자가 어떤 권한 검증을 완료해야 하는지 주석이나 메서드명으로 드러낸다.
- 다른 도메인의 상태를 변경하는 공개 Service는 Command 객체를 입력으로 받는다.
- 조회 조합이 필요한 경우에도 다른 도메인 Repository를 직접 호출하지 않는다.

예시:

```txt
resource.service.ResourceAccessService
  getReadableResource(userId, resourceId)
  validateReadable(userId, resourceId)

project.service.RoomAccessService
  validateRoomMember(userId, roomId)
  getRoomMemberRole(userId, roomId)

work.task.service.TaskDisplayService
  getDashboardTasks(userId)
  getWidgetTasks(userId, roomId)
```

도메인 공개 Service 이름은 기능이 아니라 외부에 제공하는 계약을 기준으로 정한다. 예를 들어 `TaskService` 전체를 열기보다 `TaskDisplayService`, `TaskApprovalService`처럼 필요한 계약만 열 수 있다.

agent 연결 규칙:

```txt
허용:
resource.service -> agent.service.AgentRequestService
chat.service -> agent.service.AgentCommandService
work.task.service / work.wbs.service -> agent.service.AgentSuggestionService
personal.suggestion.service -> agent.service.PersonalAgentService

금지:
resource.service -> agent.rag.*
chat.service -> agent.prompt.*
work.task.service -> agent.repository.*
personal.suggestion.service -> agent.entity.*
```

agent가 생성한 후보를 확정 데이터로 반영하는 책임은 target 도메인 Service가 가진다.

agent 연결은 아래 순서를 기본으로 한다.

```txt
1. 호출 도메인 Service가 사용자 인증과 도메인 권한을 검증한다.
2. 호출 도메인 Service가 agent에 넘길 context 범위를 제한한다.
3. 호출 도메인 Service가 agent 공개 Service 요청 DTO를 만든다.
4. agent 공개 Service는 후보 생성, 모델 호출, Structured Output 검증을 수행한다.
5. agent는 생성된 모든 후보를 `agent_suggestions`에 DRAFT 상태로 저장한다.
6. agent는 저장된 후보 ID, 상태, 실패 사유 같은 실행 결과만 반환한다.
7. 확정 데이터 저장은 사용자 승인 후 target 도메인 Service가 처리한다.
```

agent 공개 Service 요청 DTO에는 Entity를 넣지 않는다. `Resource`, `ProjectRoom`, `Task`, `ChatMessage`, `Memo` 같은 Entity 대신 필요한 값만 담은 Command, Query, Context, Result 객체를 사용한다.

agent 연결에 사용하는 Command, Query, Context, Result는 각 도메인의 `dto` 또는 `service` 하위에 둔다. 예를 들어 `resource`가 agent에 자료 분석을 요청할 때 필요한 값은 `resource` 쪽에서 `ResourceAnalysisContext` 또는 `RequestResourceAnalysisCommand` 같은 값 객체로 만들고, `agent`는 Entity가 아니라 그 값 객체만 입력으로 받는다.

agent 공개 Service는 다른 도메인의 내부 구현을 알면 안 된다. 예를 들어 `AgentRequestService`는 `ResourceRepository`를 직접 호출하지 않고, 자료 본문이 필요하면 `resource.service`가 권한 확인 후 추출한 텍스트와 메타데이터를 요청 DTO에 담아 전달한다.

agent가 다른 도메인 데이터를 다시 조회해야 하는 구조는 만들지 않는다. 필요한 데이터가 부족하면 agent 내부에서 Repository를 찾는 대신, 공개 Service 요청 DTO를 보강하거나 호출 도메인 Service에 context 준비 책임을 추가한다.

예시:

```txt
자료 업로드 분석:
resource.service
-> 자료 접근 권한 확인
-> 텍스트 추출 결과와 자료 메타데이터 context 생성
-> AgentRequestService
-> agent가 AI 분석 대상 문서 여부 분류
-> 필요한 자료만 ai_documents에 저장
-> ai_documents.resource_id에 포함된 resources 기준으로 chunk 추출과 embedding 생성
-> 관련 자료 검색/RAG context 구성
-> agent 분석 결과 반환
-> agent.suggestion이 분석에서 나온 후보를 agent_suggestions에 저장
-> resource.service가 resource_summaries와 ai_documents 상태 저장 여부를 결정

프로젝트룸 채팅 명령어:
chat.service
-> chat_room_members / room_members 권한 확인
-> 최근 메시지와 room_memory_summaries를 제한 조회
-> AgentCommandService
-> agent 응답 후보 반환
-> chat.service가 AGENT_RESPONSE 저장

TODO 후보 승인:
work.task.service
-> agent.suggestion이 관리하는 agent_suggestions 후보 조회
-> 사용자 수정값과 work 도메인 규칙 검증
-> work.task.service가 tasks 생성
```

도메인별 agent 연결 책임:

| 호출 도메인 | agent에 넘겨도 되는 것 | agent에 넘기면 안 되는 것 | 확정 처리 책임 |
| --- | --- | --- | --- |
| `resource` | 권한 확인된 자료 텍스트 일부, 파일명, mime type, visibility, roomId | 파일 원본 경로, 다운로드 URL, PERSONAL 자료를 프로젝트룸 context에 포함한 값 | `resource` |
| `project` | roomId, 사용자 역할, 프로젝트룸 설정 중 필요한 값 | room_members 전체 목록, 권한 확인 전 roomId만 넘기는 흐름 | `project` 또는 호출 도메인 |
| `work` | 후보 생성을 위한 task/wbs 요약, 담당자 후보, 일정 범위 | `tasks`, `wbs_items`, `schedules` Entity | `work.task`, `work.wbs`, `work.schedule` |
| `chat` | 제한된 최근 메시지, room memory summary, 명령어 내용 | 전체 채팅방 원문, 1:1 친구 채팅을 agent로 넘기는 흐름 | `chat` |
| `personal` | 사용자가 승인한 개인 context, 개인 제안 표시 상태 | 개인 에이전트 원문 대화 서버 저장, 승인 없는 메모/일정 확정 | `personal` 또는 대상 도메인 |
| `memory` | 승인 대상 요약 후보, 출처 정보, 사용자 수정값 | 개인 에이전트 원문 대화, 미승인 요약을 확정 기억으로 저장하는 흐름 | `memory` |

## 10. 데이터 저장 규칙

### 10.1 AI 문서와 임베딩

원본 자료는 `resources`가 소유한다. 에이전트는 `resources` 전체를 바로 임베딩하지 않고, 권한 확인이 끝난 자료 중 업무 후보 생성, 요약, 맥락 유지에 쓸 만한 문서만 분류해 `ai_documents`에 저장한다.

저장 기준:

- `resources`: 사용자가 업로드하거나 동기화한 자료 원본과 접근 권한 기준
- `ai_documents`: AI가 읽고 분석할 대상으로 선별된 자료 목록. `resource_id`로 원본 자료를 연결한다.
- `resource_embeddings`: 실제 chunk와 embedding 저장. 원본 `resource_id` 기준으로 저장하되, 임베딩 대상은 `ai_documents.resource_id`에 포함된 `resources`로 제한한다.
- `resource_summaries`: 자료 단위 요약
- `agent_suggestions`: 승인 또는 확인이 필요한 후보와 제안

에이전트 분석 파이프라인:

```txt
resources
-> 권한 확인과 텍스트 추출
-> AI 분석 대상 분류
-> ai_documents 저장
-> chunk 추출
-> resource_embeddings 저장
-> 관련 자료 검색/RAG context 구성
-> resource_summaries 저장
-> agent_suggestions에 후보/제안 DRAFT 저장
```

`ai_documents`에 저장되지 않은 자료는 AI 분석, 임베딩, RAG 후보 생성의 기본 입력으로 사용하지 않는다. 사용자가 나중에 분석을 요청하거나 자료 상태가 바뀌면 다시 분류할 수 있다.

### 10.2 후보와 확정 데이터

에이전트가 만든 값은 기본적으로 후보 데이터다.

에이전트가 생성한 후보 데이터는 성격에 따라 저장 위치를 나누지 않고 `agent_suggestions`에 통합 저장한다. 요구사항, TODO, WBS, 일정, 확인 질문, 계약 추출값, 문서 검토 항목, 문서 초안, 하루정리, 메모 제안은 `suggestion_type`과 `payload_json`으로 구분한다. 자료 기반 제안은 `agent_suggestions.resource_id`로 원본 자료를 직접 연결하고, 세부 대상 정보와 원문 근거는 `payload_json`에 둔다. agent는 후보 내용을 생성하고 `agent_suggestions`에 저장하지만 확정 데이터는 직접 만들지 않는다.

후보 데이터 예시:

```txt
agent_suggestions.suggestion_type = REQUIREMENT
agent_suggestions.suggestion_type = TODO
agent_suggestions.suggestion_type = WBS
agent_suggestions.suggestion_type = SCHEDULE
agent_suggestions.suggestion_type = QUESTION
agent_suggestions.suggestion_type = CONTRACT_FIELD
agent_suggestions.suggestion_type = CONTRACT_REVIEW
agent_suggestions.suggestion_type = DOCUMENT_DRAFT
agent_suggestions.suggestion_type = DAILY_SUMMARY
agent_suggestions.suggestion_type = MEMO
```

통합 저장 기준:

- 에이전트가 만든 후보 원본은 항상 `agent_suggestions`에 저장한다.
- 후보의 종류는 `suggestion_type`으로 구분한다.
- 자료 기반 후보는 `agent_suggestions.resource_id`로 원본 자료를 연결한다.
- 후보의 대상 도메인, 대상 ID, 원문 근거, field_key 같은 상세 정보는 `payload_json` 안에 넣는다.
- `resource_summaries`는 자료 단위 요약을 저장한다.
- `ai_documents`는 자료의 AI 문서 분류와 분석 상태를 저장한다.
- 승인/보류/거절이 필요한 후보 원본은 `agent_suggestions`에 둔다.
- `tasks`, `wbs_items`, `schedules`, `memos`, `project_rooms`, `room_memory_summaries`, `daily_summaries`는 승인 후 생성되거나 수정되는 확정 테이블이다.

확정 데이터:

```txt
project_rooms
wbs_items
tasks
schedules
memos
time_logs
project room payment/reference fields
room_memory_summaries
daily_summaries
```

규칙:

- 후보는 사용자가 승인하기 전까지 확정 데이터로 쓰지 않는다.
- 승인된 후보만 관련 도메인 Service가 확정 데이터로 반영한다.
- agent 모듈은 확정 데이터를 직접 생성하지 않는다.
- agent 모듈은 후보를 `agent_suggestions`에 저장하고, 승인 후 target 도메인 Service가 확정 저장을 맡는다.

### 10.3 원본 데이터 책임과 중복 저장 금지

여러 화면에서 같은 데이터를 보여주더라도 원본은 하나의 도메인만 소유한다. 다른 도메인은 원본 데이터를 복사 저장하지 않고, 필요한 시점에 원본 도메인의 조회 결과를 사용한다.

```txt
자료 원본: resource
파일 원본: resource + storage
채팅 원본: chat
TODO 원본: work.task
WBS 원본: work
일정 원본: schedules / work.schedule
캘린더 화면 조합: personal.calendar
메모 원본: memos
타이머와 시간 기록 원본: time_logs
하루정리 요약 원본: memory
프로젝트룸 장기기억 원본: memory
위젯 context, 버블 설정, 항목 상태와 사용 집계 원본: widget
알림 원본: personal.notification
보이스 상태 원본: voice
```

규칙:

- dashboard와 widget은 원본 데이터를 복사 저장하지 않고 조회 결과를 조합한다.
- 개인 TODO와 프로젝트룸 TODO 상태 변경은 work.task에서만 처리한다.
- WBS 상태 변경은 work에서만 처리한다.
- 일정 변경은 `schedules`를 기준으로 `work.schedule`에서 처리한다. 개인 일정은 `room_id`가 없고, 프로젝트룸 일정은 `room_id`가 있다.
- personal.calendar는 Google Calendar에서 가져온 외부 일정과 Bubli `schedules` 일정을 같은 캘린더 화면에 조합해 표시한다. Google Calendar에서 가져온 일정은 외부 일정 레이어이고, Bubli가 생성한 일정 원본은 `schedules`다.
- 메모 변경은 `memos` 기준으로 처리한다. 개인 메모는 본인 `user_id`, 프로젝트룸 메모는 `room_members` 권한을 확인한다.
- 타이머와 작업 시간 변경은 `time_logs` 기준으로 처리한다. 작업 타이머는 프로젝트룸 context와 연결될 수 있다.
- 채팅 메시지 저장과 순서 관리는 chat에서만 처리한다.

### 10.4 파일 저장

파일 저장은 `StorageService` 인터페이스로 감싼다.

```txt
StorageService
  save(file)
  getDownloadUrl(storageKey)
  delete(storageKey)
```

구현체:

```txt
LocalFileStorageService
S3StorageService
```

규칙:

- S3 객체는 public으로 열지 않는다.
- 다운로드는 서버 권한 확인 후 URL을 발급한다.
- 파일 크기 제한과 분석 텍스트 길이 제한은 별도로 둔다.
- 같은 파일 checksum은 반복 분석하지 않는다.
- 저장 용량을 서버 DB에 기록해야 하는 경우에는 별도 원본 테이블을 DB 설계 문서에서 확정한 뒤 관리한다.

### 10.5 Tauri SQLite 저장 기준

채팅 원본은 서버 DB의 `chat_messages`다.

Tauri SQLite에는 서버 원본이 아니라 로컬 기능, 캐시, 개인 원문 데이터를 저장한다.

SQLite 저장 대상:

|대상|저장 이유|서버 반영 기준|
|---|---|---|
|`local_files`|개인 관리 폴더의 파일 색인|개인 자료함 동기화가 켜진 항목은 서버 개인 자료함에 반영|
|`local_file_events`|파일 추가, 수정, 삭제, 이동 이벤트 기록|개인 자료함 동기화가 켜진 경우 localsync API로 서버 반영|
|`sync_operation_logs`|되돌리기와 충돌 확인용 로컬 작업 로그|기본적으로 서버 원본 아님|
|`local_room_message_cache`|프로젝트룸 최근 채팅 캐시|서버 `chat_messages`가 원본, 손상 시 서버에서 재생성|
|`local_room_cache_state`|채팅 캐시의 `room_sequence` 동기화 상태|서버 `chat_messages` 기준으로 보정|
|`local_agent_messages`|개인 에이전트 최근 원문 대화|서버 저장 금지|
|`local_agent_summaries`|개인 에이전트 로컬 요약|사용자가 확인한 최종 요약만 서버 `daily_summaries`에 저장|
|`local_widget_display_cache`|버블에 빠르게 표시할 최근 데이터 캐시|서버 원본 데이터 기준으로 재생성 가능|
|`local_widget_usage_events`|버블 열기, 닫기, 클릭, 머문 시간 같은 상세 이벤트|원문 이벤트는 서버 저장 금지|
|`local_widget_usage_rollups`|서버 전송 전 위젯 사용 집계|집계 결과만 서버 `widget_daily_summaries`에 반영|
|`local_timer_state`|앱 비정상 종료 후 타이머 복구|복구 후 확정 작업 시간만 서버에 반영|
|`local_sync_outbox`|미전송 요청 재전송 대기열|`idempotency_key`로 중복 전송 방지 후 서버 반영|
|`local_backup_manifest`|SQLite 백업 파일 목록과 무결성 정보|기본적으로 사용자 기기 안에서만 사용|

규칙:

- 채팅 메시지는 DB 저장 성공 후 WebSocket으로 전송한다.
- `client_message_id`로 중복 저장을 막는다.
- `room_sequence`로 채팅방 메시지 순서를 관리한다.
- Tauri 채팅 캐시는 서버 원본을 복제한 값이며 원본이 아니다.
- 프로젝트룸 채팅 캐시는 최근 100개 메시지를 기본 기준으로 둔다.
- 위젯 항목 상태의 원본은 서버 `widget_item_states`다.
- 위젯 날짜별 사용 집계의 원본은 서버 `widget_daily_summaries`다.
- 위젯 상세 사용 이벤트 원문은 Tauri SQLite에만 저장한다.
- 위젯 표시 캐시는 서버 원본 데이터를 빠르게 보여주기 위한 로컬 캐시이며 원본이 아니다.
- 활동 감지 정보는 사용자 동의가 켜진 경우에만 기록한다.
- 개인 에이전트 원문은 서버 DB에 저장하지 않는다.
- 개인 에이전트 최근 대화 원문은 Tauri SQLite의 `local_agent_messages`에만 저장한다.
- 사용자가 확인한 하루정리 요약만 서버 `daily_summaries`에 저장한다.

## 12. 테스트와 API 검증

테스트 우선순위:

```txt
1. 권한과 격리
2. 후보 생성과 승인
3. 자료와 파일 저장
4. 채팅과 알림
5. 위젯과 대시보드
```

필수 테스트:

- PERSONAL 자료는 owner만 조회할 수 있다.
- ROOM_SHARED 자료는 room_members만 조회할 수 있다.
- 프로젝트룸 에이전트는 PERSONAL 자료를 읽지 않는다.
- 후보 승인 전에는 tasks/wbs_items가 생성되지 않는다.
- 후보 승인 후에만 확정 데이터가 생성된다.
- agent가 생성한 요구사항, TODO, WBS, 확인 질문, 계약 추출값 후보는 모두 `agent_suggestions`에 저장된다.
- agent 공개 Service 요청 DTO에 다른 도메인 Entity를 넣지 않는다.
- agent 실행 흐름에서 다른 도메인 Repository를 직접 호출하지 않는다.
- agent 입력 context는 호출 도메인 Service의 권한 검증 이후에만 생성된다.
- agent 결과 승인 시 실제 저장은 resource, work, personal, memory, chat 같은 target 도메인 Service에서 처리한다.
- widget summary는 원본 데이터를 복사 저장하지 않는다.
- widget summary는 접근 권한이 없는 `selected_room_id`의 데이터를 반환하지 않는다.
- chat_messages는 client_message_id 중복 저장을 막는다.
- Tauri 채팅 캐시는 서버 메시지로 복구 가능하다.
- 개인 에이전트 원문은 서버 DB에 저장하지 않는다.

테스트 작성 기준:

- Service 테스트는 권한, 상태 전이, 중복 방지, 원본 데이터 소유 규칙을 우선 검증한다.
- Controller 테스트는 응답 형식, validation 실패, 인증 실패를 우선 검증한다.
- Repository 테스트는 복잡한 JPQL, QueryDSL, custom repository에 한해 작성한다.
- agent 관련 테스트는 후보 생성 결과가 확정 테이블을 직접 변경하지 않는지 확인한다.
- agent 관련 테스트는 PERSONAL 자료와 개인 에이전트 원문 대화가 프로젝트룸 agent context에 포함되지 않는지 확인한다.
- WebSocket은 DB 저장 성공 후 이벤트가 나가는 흐름을 검증한다.
- 외부 연동은 테스트 대역을 사용하고 실제 S3, LiveKit, 모델 API를 단위 테스트에서 호출하지 않는다.
- 권한 테스트는 성공 케이스보다 차단 케이스를 먼저 작성한다.

Testcontainers 기준:

- PostgreSQL, pgvector, Flyway, JPA 매핑까지 같이 확인해야 하는 테스트는 Testcontainers를 사용한다.
- 공통 베이스는 `src/test/java/com/bubli/support/PostgresIntegrationTestSupport.java`를 사용한다.
- API 통합 테스트는 `MockMvc`로 실제 Security filter와 `@CurrentUser` 흐름을 통과하게 만든다.
- 인증이 필요한 API는 테스트 안에서 `JwtTokenProvider`로 access token을 만들고 `Authorization: Bearer ...` 헤더를 넣는다.
- Docker가 꺼진 로컬 환경에서는 `@Testcontainers(disabledWithoutDocker = true)` 기준으로 스킵될 수 있다. Docker를 켠 환경에서는 같은 테스트가 PostgreSQL 컨테이너로 실행된다.
- 단순 Service 규칙은 Mockito 단위 테스트로 빠르게 검증하고, DB 제약, Flyway schema, 권한 필터, 공통 응답 형식까지 묶인 흐름은 Testcontainers 통합 테스트로 검증한다.

개발 중 API 수동 검증은 IntelliJ IDEA의 HTTP Client를 사용한다. 팀에서 공유할 요청 예시는 `.http` 파일로 저장한다.

권장 위치:

```txt
docs/http/
  auth.http
  project-rooms.http
  resource.http
  personal.http
  work.http
  agent.http
  chat.http
  memory.http
  widget.http
  voice.http
  localsync.http
```

규칙:

- `.http` 파일은 개발 중 API 동작 확인과 팀 공유용이다.
- 인증이 필요한 API는 로그인 요청과 토큰 변수를 함께 둔다.
- 로컬, 개발, 운영 환경 baseUrl은 변수로 분리한다.
- 비밀번호, JWT, API key 같은 민감정보는 `.http` 파일에 직접 저장하지 않는다.
- `.http` 파일은 자동화 테스트를 대체하지 않는다.

## 13. 개발 순서 기준

팀 개발은 공통 기반을 먼저 맞춘 뒤 도메인 기능을 병렬로 구현한다.

권장 순서:

```txt
1. global.response, global.error, global.security, global.trace, global.validation
2. auth, user
3. project, resource, work
4. chat, voice
5. agent 실행 요청과 후보 저장
6. memory, widget, localsync, activity
7. .http 예시, 권한 테스트, 통합 흐름 검증
```

기준:

- 공통 응답과 에러 형식이 확정되기 전에는 도메인 Controller를 대량으로 만들지 않는다.
- 인증 사용자 주입 방식이 확정되기 전에는 userId를 request body로 받는 API를 만들지 않는다.
- project와 resource 권한 검증 기준이 먼저 잡힌 뒤 agent, widget, chat에서 해당 데이터를 참조한다.
- agent는 처음부터 분리 가능성을 고려하되, 초기 구현은 백엔드 내부 모듈로 둔다.
- DB 상세 설계가 확정되지 않은 테이블은 임시 Entity로 먼저 만들지 않는다.
- API 명세서와 데이터 딕셔너리가 확정된 뒤 `.http` 예시를 맞춘다.

## 14. 금지사항

공통 금지:

- Controller에 비즈니스 로직을 작성하지 않는다.
- Controller에서 Repository를 직접 호출하지 않는다.
- Entity를 API 응답으로 직접 반환하지 않는다.
- 다른 도메인의 Repository를 직접 호출하지 않는다.
- 공통 응답 형식 밖의 임의 응답 구조를 만들지 않는다.
- traceId 없는 에러 응답을 만들지 않는다.
- enum 상태값 대신 임의 문자열을 사용하지 않는다.
- 사용자 권한 검증 없이 roomId, resourceId 기준 데이터를 반환하지 않는다.

자료와 권한 금지:

- PERSONAL 자료를 room_members에게 노출하지 않는다.
- 프로젝트룸 에이전트 context에 PERSONAL 자료를 넣지 않는다.
- S3 객체를 public으로 열지 않는다.
- 다운로드 URL을 권한 확인 없이 발급하지 않는다.
- 개인 자료함 동기화가 꺼진 로컬 파일을 서버 자료로 등록하지 않는다.
- 개인 자료를 프로젝트룸에 자동 공유하지 않는다.

에이전트 금지:

- agent 모듈에서 `tasks`, `wbs_items`, `schedules` 확정 데이터를 직접 생성하지 않는다.
- agent 모듈에서 다른 도메인의 Repository나 Entity를 직접 사용하지 않는다.
- 에이전트 결과를 사용자 승인 없이 확정 데이터로 반영하지 않는다.
- LLM 출력 JSON을 검증 없이 저장하지 않는다.
- 문서 원문 전체나 개인 에이전트 원문 대화를 로그에 저장하지 않는다.
- 에이전트가 파일 삭제, 파일 공유, 외부 전송, 법률 판단 확정을 수행하지 않는다.
- 에이전트가 권한 검증 전의 resourceId, roomId, filePath만으로 직접 context를 조회하지 않는다.
- 에이전트 공개 Service 요청 DTO에 다른 도메인 Entity를 담지 않는다.

위젯 금지:

- widget 모듈이 TODO, 일정, 메모, 타이머 원본 데이터를 소유하지 않는다.
- widget 테이블에 각 버블의 원본 데이터를 복사 저장하지 않는다.
- 위젯 상세 사용 이벤트 원문을 서버 DB에 저장하지 않는다.
- 접근 권한이 없는 프로젝트룸 데이터를 위젯 summary에 포함하지 않는다.
- 비활성화된 버블의 상세 데이터를 불필요하게 조회하지 않는다.

Tauri와 개인정보 금지:

- 개인 에이전트 원문 대화를 서버 DB에 저장하지 않는다.
- 전체 PC 파일 목록을 자동 색인하지 않는다.
- 사용자가 선택하지 않은 폴더에 접근하지 않는다.
- 화면 전체 내용과 키보드 입력을 수집하지 않는다.
- 활동 감지 동의가 꺼진 사용자에게 앱/창 제목 기록을 저장하지 않는다.
- `local_sync_outbox` 재전송 요청을 `idempotency_key` 없이 서버에 반영하지 않는다.
