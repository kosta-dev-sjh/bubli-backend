# GitHub Issues & Projects 보드 완전 가이드

## 1. GitHub Issues란?

레포지토리에 딸린 **할 일 목록**이다. "이 기능 만들어야 해", "여기 버그 있어", "이 설정 바꿔야 해" 같은 작업 단위를 글로 남기고, 담당자를 정하고, 진행 상태를 추적하는 도구.

---

## 2. 이슈를 왜 쓰는가

| 장점 | 설명 |
|------|------|
| **작업이 사라지지 않는다** | 카톡이나 구두로 "이거 해줘"라고 하면 묻힌다. 이슈로 남기면 닫히기 전까지 계속 보인다. |
| **누가 뭘 하고 있는지 보인다** | 각 이슈에 담당자를 지정하면 팀원 5명이 지금 뭘 맡고 있는지 한눈에 파악된다. |
| **코드와 연결된다** | PR이나 커밋에서 이슈 번호를 달면, 어떤 코드가 어떤 작업을 위해 생겼는지 추적 가능하다. |
| **진행률이 보인다** | 열린 이슈 = 남은 일, 닫힌 이슈 = 끝난 일. "우리 몇 퍼센트 했지?"를 숫자로 볼 수 있다. |

---

## 3. 이슈 작성법

### 3-1. 이슈 만드는 순서

1. 레포 → **Issues** 탭 클릭
2. 초록색 **New issue** 버튼 클릭
3. 이슈 템플릿이 있으면 선택 (bug_report / feature_request), 없으면 "Open a blank issue"
4. 제목과 본문 작성
5. 오른쪽 사이드바에서 라벨, 담당자, 마일스톤 설정
6. **Submit new issue** 클릭

### 3-2. 제목 쓰는 법

짧고 구체적으로. 뭘 해야 하는지가 제목만 봐도 보여야 한다.

- 나쁜 예: `인증 관련`, `버그 수정`
- 좋은 예: `JWT 로그인 API 구현`, `프로젝트룸 생성 시 기본 룸 자동 생성 안 되는 버그`

### 3-3. 본문 쓰는 법

```markdown
## 작업 내용
Google 로그인 후 JWT 발급 흐름을 구현한다.
- POST /api/auth/google/login
- Google 계정으로 사용자 식별
- 액세스 토큰 + 리프레시 토큰 발급

## 참고
- 백엔드 개발 가이드 8.1 auth 섹션 참고
- Spring Security + jjwt 라이브러리 사용

## 완료 기준
- [ ] 로그인 성공 시 토큰 반환
- [ ] Google 인증 정보가 유효하지 않으면 401 응답
- [ ] 최초 로그인 사용자는 users 기준에 맞게 생성 또는 연결
```

### 3-4. 라벨 (Labels)

이슈 종류를 색깔 태그로 분류한다. 팀에서 미리 만들어 두면 편하다.

| 라벨 | 용도 |
|------|------|
| `feature` | 새 기능 개발 |
| `bug` | 버그 수정 |
| `chore` | 설정, 문서, 빌드 |
| `refactor` | 리팩토링 |
| `1순위` | 필수 (MVP) |
| `2순위` | 중요하지만 MVP 이후 |
| `backend` | 백엔드 작업 |
| `frontend` | 프론트엔드 작업 |

### 3-5. 담당자 (Assignees)

오른쪽 사이드바 → Assignees → 팀원 선택. 한 이슈에 여러 명도 가능하지만, **책임자는 1명**으로 정하는 게 좋다.

### 3-6. 마일스톤 (Milestones)

마일스톤 = 기한이 있는 큰 목표. 예를 들어:

- `1차 MVP (~ 6/27)` — 인증, 프로젝트룸, 자료 업로드, 기본 채팅
- `2차 AI 고도화 (~ 7/4)` — RAG, 에이전트, 위젯
- `3차 통합 (~ 7/8)` — Tauri, 보이스, 최종 QA

이슈를 마일스톤에 넣으면, 마일스톤 화면에서 진행률 바가 보인다.

### 실제 예시

> **제목**: `[feature] JWT 로그인/회원가입 API 구현`
>
> **라벨**: `feature`, `backend`, `1순위`
>
> **담당자**: 김미연
>
> **마일스톤**: 1차 MVP (~ 6/27)

## 4. 이슈와 PR 연결법

### 4-1. 커밋 메시지에서 참조

커밋 메시지에 `#이슈번호`를 쓰면 해당 이슈 타임라인에 커밋 링크가 자동으로 달린다.

```
feat: JWT 로그인 API 구현 #3
```

이슈 #3 페이지에 가보면 "이 커밋이 이 이슈를 참조했다"고 기록이 남는다.

### 4-2. PR에서 이슈 자동 닫기

PR 본문에 아래 키워드 + `#번호`를 쓰면, **PR이 머지될 때 해당 이슈가 자동으로 닫힌다.**

- `closes #3`
- `fixes #3`
- `resolves #3`

**예시 — PR 본문:**

```markdown
## 작업 내용
JWT 로그인/회원가입 API를 구현했습니다.

closes #3
```

이 PR이 develop에 머지되면 → 이슈 #3이 자동으로 "Closed" 상태가 된다.

### 4-3. 왜 연결하나?

- 나중에 "로그인 기능 어떻게 만들었지?" → 이슈 #3 → 연결된 PR → 실제 코드를 바로 볼 수 있다
- 발표 때 "이슈 20개 중 18개 닫혔습니다" 하면 진행률이 눈에 보인다

---

## 5. GitHub Projects란?

이슈들을 **칸반 보드**(Todo / In Progress / Done) 형태로 시각적으로 관리하는 도구. 이슈 탭이 "목록"이라면, Projects는 그 목록을 **"보드판에 붙인 포스트잇"** 처럼 보여준다.

---

## 6. 프로젝트 보드 만드는 법

1. GitHub에서 레포 → **Projects** 탭 클릭
2. 초록색 **New project** 클릭
3. 템플릿 선택: **Board** (칸반) 선택
4. 프로젝트 이름 입력: 예) `Bubli Backend Sprint Board`
5. **Create project** 클릭

만들어지면 기본 컬럼 3개가 보인다:

| 컬럼 | 의미 |
|------|------|
| **Todo** | 아직 시작 안 한 일 |
| **In Progress** | 지금 작업 중인 일 |
| **Done** | 끝난 일 |

필요하면 컬럼을 추가할 수 있다:
- `Review` — PR 올려놓고 리뷰 기다리는 중
- `Blocked` — 다른 작업이 끝나야 진행 가능

---

## 7. 이슈와 보드 연결 방법

### 방법 1: 보드에서 이슈 추가

1. 프로젝트 보드 열기
2. Todo 컬럼 하단의 **+ Add item** 클릭
3. `#`을 누르면 레포의 이슈 목록이 뜬다
4. 추가할 이슈를 선택하면 카드로 들어간다

### 방법 2: 이슈 페이지에서 프로젝트 연결

1. 이슈 상세 페이지 → 오른쪽 사이드바 → **Projects** 섹션
2. 프로젝트 보드를 선택
3. Status를 "Todo"로 설정

### 방법 3: 이슈 만들 때 바로 연결

새 이슈를 만들 때 오른쪽 사이드바에서 Projects를 선택하면 생성과 동시에 보드에 올라간다.

### 자동화 (워크플로)

프로젝트 보드 설정 → **Workflows**에서:

- **Item added to project** → Status를 "Todo"로 자동 설정
- **Pull request merged** → Status를 "Done"으로 자동 이동
- **Item closed** → Status를 "Done"으로 자동 이동

이걸 켜 두면 이슈를 추가하면 자동으로 Todo에 들어가고, PR 머지로 이슈가 닫히면 자동으로 Done으로 이동한다.

---

## 8. 보드 칸반 사용법 — 하루 흐름

**아침 — 할 일 확인:**
보드를 열어서 Todo에 있는 카드 중 오늘 할 것을 In Progress로 드래그한다.

**작업 시작:**
```bash
git switch -c feature/auth-jwt
```

**작업 중:**
카드가 In Progress에 있으므로 팀원들이 "아 얘 지금 이거 하고 있구나" 알 수 있다.

**PR 올릴 때:**
PR 본문에 `closes #3` 쓰고 develop에 PR을 올린다.

**머지되면:**
워크플로가 켜져 있으면 자동으로 Done으로 이동.

```
Todo → In Progress → (Review) → Done
 작업 시작    PR 올림    머지 완료
```

## 9. 팀 운영 팁

### 이슈를 언제 만드나?

기획 단계에서 기능 목록이 나오면 그걸 이슈로 쪼갠다. 예: "auth 모듈" 하나가 아니라, `회원가입 API`, `로그인 API`, `토큰 갱신 API`로 쪼갠다. **한 이슈 = PR 1~2개로 끝낼 수 있는 크기**가 좋다.

### 일일 스탠드업 대용

매일 아침 팀 단톡에 "보드 확인하고 오늘 할 거 In Progress로 옮겨줘" 한 마디면 된다. 별도 회의 없이도 보드만 보면 전체 진행 상황이 보인다.

### 마일스톤으로 주간 목표 관리

마일스톤 `1차 MVP (~ 6/27)`에 이슈 10개를 넣어두면 → 마일스톤 페이지에서 `7/10 closed` 이런 식으로 진행률 바가 보인다.

### 보드에서 필터링

- `label:backend` → 백엔드 이슈만 보기
- `assignee:hongjunhwa` → 내 이슈만 보기
- `milestone:"1차 MVP"` → 이번 스프린트 이슈만 보기

### 우리 팀(5인) 추천 운영 방식

1. 기획 끝나면 → 기능별로 이슈를 만든다 (20~30개 정도)
2. 라벨 → `feature`/`bug`/`chore` + `backend`/`frontend` + `1순위`/`2순위`
3. 마일스톤 → 1차 MVP / 2차 AI / 3차 통합
4. 담당자 → 각 이슈에 1명씩
5. 보드 → 매일 아침 확인, 작업 시작하면 In Progress로 옮기기
6. PR → `closes #번호` 습관화 → 자동으로 Done 이동

---

## 한 줄 요약

**이슈 = 할 일 카드, 보드 = 그 카드를 붙이는 벽, PR의 `closes #번호` = 카드를 자동으로 완료로 옮기는 마법.**
